[00:00]  1. OPENING AND PROBLEM STATEMENT
==============================================================================
SAY:
"Good day, and thank you for your time. This is my submission for AtlasCare, an
agentic AI customer-support system built for a retail organisation, Acme Retail Co.

I would like to begin with the question that shaped the entire design. A customer
submits a single instruction — for example, 'cancel my laptop and refund me
ninety-five thousand rupees' — and an autonomous agent decides whether, and how, to
act on it. The decisive question is not whether the model will occasionally be
wrong, but what happens when it is: when it misreads an order, when it is
manipulated by a crafted message, or when the payment gateway fails midway through a
disbursement.

Most demonstrations never confront that question, because in a demonstration the
model is always correct. This system was designed on the opposite assumption — that
the model will sometimes be wrong about matters that involve money and customer
data.

Accordingly, the system was not optimised to answer a single question elegantly. It
was optimised for the operational reality of an on-call engineer at three in the
morning, asking one question: did the system disburse funds that it should not have,
or fail to disburse funds that it should have?

Every design decision follows from a single governing principle, which recurs
throughout this presentation: the model may propose, but deterministic code makes
the final decision. Behind every rule that touches money or mutates an order sits a
plain-Python guardrail, fully tested, that does not depend on the model having obeyed
its instructions. The prompt is the first line of defence; it is never the last. As
the remainder of this presentation will demonstrate, nearly every such guardrail was
introduced in response to a specific failure that surfaced during adversarial
testing."

==============================================================================
[01:30]  2. SYSTEM ARCHITECTURE
==============================================================================
SHOW: agent/graph.py — build_graph(); or the rendered graph on the admin dashboard.

SAY:
"The platform comprises three components: a FastAPI application exposing the HTTP API
in main.py; a Gradio web application providing the customer chat interface; and the
agent itself, implemented as a LangGraph state machine in agent/graph.py. The
language model is accessed through an OpenAI-compatible client, which permits the
entire model provider to be changed through environment variables alone, with no code
modification.

The central architectural decision is that a customer request does not execute within
an open-ended loop in which the model invokes tools until it decides it is finished.
It proceeds through an explicit, inspectable graph of named stages, and the edges
between those stages are themselves deterministic code. The request path is as
follows:

  input redaction → confirmation check → pre-guardrail → policy grounding
      → planner → tool executor → post-guardrail → responder → evaluator → end

Each node serves a defined purpose. Input redaction is the true entry point: it
deterministically masks payment card numbers, security codes, email addresses, and
telephone numbers before the message enters the conversation history, reaches the
model, or is written to any log. Confirmation check resolves any pending
confirmation from the previous turn before new work begins. The pre-guardrail applies
deterministic policy before the model is consulted — over-limit refunds,
fraud and safety reports, and malformed identifiers. Policy grounding separates
question-answering from action-taking: a general policy question is answered from the
knowledge base and bypasses the planner entirely. The planner selects tools, using a
two-tier model strategy. The tool executor is the single chokepoint through which
every action passes. The post-guardrail is the final monetary safety net after
execution. The responder composes the customer-facing reply exclusively from verified
tool results. The evaluator is an LLM-as-judge pass permitting at most one
regeneration.

Beneath the graph, the layering is strict: tools call services, services call
repositories, and repositories read and write the data files. Monetary values are
handled exclusively with Python's Decimal type, never floating point."

SHOW: observability/tracer.py and the admin dashboard.

SAY:
"Observability is a first-class concern. Every request is assigned a trace identifier
on arrival; every tool invocation and guardrail decision is recorded with its status
and latency; and the completed trace is retained in a thread-safe in-memory ring
buffer. That buffer powers the administrative dashboard, which computes the operating
KPIs directly: total requests, escalation rate, guardrail activations, ownership
denials, error counts, and latency percentiles including a count of breaches of the
three-second service target."

[04:00]  3.1  CONFIGURATION AS THE SOURCE OF TRUTH
------------------------------------------------------------------------------
SHOW: agent/guardrails.py::_resolve_auto_refund_limit ; data/payment_config.json.

SAY:
"Business policy is data-driven, so that an operator may change a rule without a code
deployment. Two values are most significant: the auto-refund limit of twenty-five
thousand rupees, and the set of supported refund methods. Each is resolved once, at
import, from payment_config.json, with an environment-variable fallback and a safe
default. The guardrail, the payment tool, and the agent prompt therefore observe a
single resolved value, with no opportunity for divergence. The supported-method set
is computed as an intersection: configuration may narrow the methods the code
supports, but never widen them. This single-sourcing was introduced after an earlier
version was found to read a configuration value in tests while ignoring it at runtime
— configuration that appeared to control behaviour but did not."

------------------------------------------------------------------------------
[05:30]  3.2  MONETARY SAFETY — THE REFUND THRESHOLD IN THREE LAYERS
------------------------------------------------------------------------------
SHOW: guardrails.py GR-001 and GR-004 ; tools/payment_tool.py::_enforce_threshold.

SAY:
"The most consequential failure this system could commit is the autonomous
disbursement of a refund that exceeds the limit and should have received human review.
This risk is therefore mitigated in three independent layers. The first, GR-001, runs
before the model: a refund request above the limit is blocked and escalated, and never
reaches the model. The second resides in the payment tool itself: immediately before
any disbursement, the amount is compared against the limit using Decimal arithmetic,
and a typed exception is raised on breach. The third, GR-004, runs after execution,
inspecting what was actually disbursed and blocking the reply on a genuine over-limit
disbursement. A single defect, or a single crafted prompt, is therefore insufficient
to release funds improperly."

------------------------------------------------------------------------------
[07:00]  3.3  CANCELLATION AND REFUND ATOMICITY
------------------------------------------------------------------------------
SHOW: agent/graph.py::_handle_cancel_item.

SAY:
"This control originated not from code review but from inspection of the data the
agent had produced. One order exhibited three cancelled items but only two refund
records: an item had been cancelled with no corresponding refund and no trace by which
to locate it. For a system that handles money, this is the unrecoverable failure mode.

The cause was one of ordering: the handler cancelled the item first and processed the
refund second, so that a gateway failure between the two steps left the cancellation
committed and the refund lost. The remedy reverses the order of operations: the refund
trace is secured first, and the cancellation is committed second. Should the refund
fail for any reason, the handler returns immediately, having modified nothing. The
worst case is thereby reduced from 'item cancelled, funds lost, no record' to 'refund
recorded, item still active' — a fully recoverable state.

A subsequent test revealed that the reordering had introduced a second defect:
cancelling an item on an order that had already shipped now disbursed the refund before
the cancellation was refused. This was corrected with a deterministic status check
performed before any funds move."

------------------------------------------------------------------------------
[09:00]  3.4  CODE-ENFORCED GUARDRAILS
------------------------------------------------------------------------------
SHOW: the unsupported-method guard, confirmation gating, and safety escalation.

SAY:
"Four decisions are deliberately withheld from the prompt. First, unsupported refund
methods: a request to refund to an unsupported instrument is detected at the message
level and answered with the supported-methods menu, with no model call. Second,
confirmation gating for high-value or ambiguous cancellations is enforced in code, in
both directions — a confirmation is imposed where the model omitted one, and a needless
confirmation is suppressed on a low-value, unambiguous item. Third, safety, fraud, and
legal reports are detected deterministically and escalated, with no autonomous action
possible on that turn. Fourth, order identifiers are normalised conservatively, such
that a malformed identifier is never silently coerced into a valid one."

------------------------------------------------------------------------------
[10:30]  3.5  SECURITY: OWNERSHIP, INJECTION, AND SANITISATION
------------------------------------------------------------------------------
SHOW: _fetch_owned_order ; the system prompt integrity block ; _sanitize_response ;
      redact_sensitive ; _UNSAFE_ADDRESS_RE ; _sanitize_case_reason.

SAY:
"Security comprises several elements. The first and most important is ownership
enforcement. Every order-touching tool retrieves the order through a single function
that verifies the order belongs to the requesting session, returning 'not found' if it
does not — so that the existence of another customer's order is never even confirmed.

The second is resistance to prompt injection: the system prompt instructs the model to
treat customer messages as data rather than instructions, and forbids disclosure of
internal identifiers, tool names, and monetary thresholds.

The third is output sanitisation: every reply is processed deterministically to rewrite
internal payment codes, remove any leaked customer identifier, and excise refund options
the system does not offer — failing safe by returning the original text if sanitisation
would corrupt it.

The fourth is input redaction, the mirror image of output sanitisation: card numbers,
security codes, email addresses, and telephone numbers are masked at the front door.
The card matcher is constrained by a Luhn checksum, so that an order total or a postal
code is never mistaken for a card number.

The fifth is stored-input sanitisation. Adversarial testing established that free-text
fields persisted to the data store — specifically the shipping-address fields and the
CRM case reason — accepted markup. An injected script tag would therefore execute when
an operator subsequently viewed the record in the administrative dashboard, a stored
cross-site-scripting exposure. Address fields now reject markup and control characters,
and case reasons are stripped of them before persistence, so that the case is still
filed while the markup is neutralised."

------------------------------------------------------------------------------
[12:30]  3.6  RESILIENCE
------------------------------------------------------------------------------
SHOW: _chat_completion_with_retry ; _safe_audit ; _append_capped.

SAY:
"Three controls address the transient failures that are inevitable at scale. Every
model call is wrapped in a retry policy with capped exponential backoff that honours a
provider's Retry-After header. The audit append is attempted up to three times and then
logged at critical severity, but never raises — a logging failure must not reverse a
completed refund. The conversation history is a bounded reducer retaining the most
recent forty messages, since a client-supplied, unbounded session identifier is
otherwise a memory-exhaustion vector."

------------------------------------------------------------------------------
[13:30]  3.7  MODEL ROUTING AND PROVIDER PORTABILITY
------------------------------------------------------------------------------
SHOW: agent/graph.py::_is_complex.

SAY:
"A single predicate routes complex, mutating, or filtering requests to a larger model,
and simple lookups to a smaller, faster one, so that routine queries complete within
the latency target while consequential requests receive the stronger model. Because the
client is OpenAI-compatible, the entire model backend can be changed through
configuration alone, providing genuine provider independence and a practical failover
path."

------------------------------------------------------------------------------
[14:00]  3.8  CATEGORY-AWARE POLICY RETRIEVAL
------------------------------------------------------------------------------
SHOW: data/kb_articles.json ; data/product_categories.json ; policy_grounding_node.

SAY:
"General policy questions are answered from the knowledge base rather than from the
model's parametric memory, because an invented policy is itself a form of harm. The
return policy, however, is not uniform across products: electronics carry a seven-day
window, apparel thirty days, footwear fourteen, beauty products are non-returnable, and
so forth, across fourteen product categories. The answer must therefore be aware of the
product.

The category taxonomy is not hardcoded; it is derived from the applies_to field of the
knowledge-base articles. At startup, each product is classified into one of those
categories by a deterministic keyword procedure — no model, and therefore reproducible
and requiring no network access; any product that cannot be classified falls into a
miscellaneous category. When a customer asks about the return window, the system
resolves the product category of the order in context and retrieves the applicable
policy. The same question consequently yields the electronics policy for a headphones
order and the apparel policy for a garment order."

[15:00]  4. ADVERSARIAL TESTING AND HARDENING
==============================================================================
SAY:
"The system was subjected to repeated adversarial testing against the live service,
using deliberately hostile, lengthy, and compound queries intended to break it. This
process was the source of most of the controls already described, and of several
further corrections, which I summarise here because the methodology is itself part of
the submission.

Three classes of finding are representative. The first concerned the refund threshold.
The prompt directed the agent to escalate any return on a delivered order whose value
exceeded the limit, but that rule existed only in the prompt: the code enforced the
threshold against the requested amount alone. A weaker model could therefore
under-state the amount — requesting twenty-four thousand against a forty-one-thousand
order — and disburse autonomously, bypassing the intended review; the same gap permitted
the full value to be drained in several sub-limit instalments. The threshold is now
enforced in code against the order value, closing both paths.

The second class concerned stored cross-site scripting, described in section 3.5: markup
persisted through the address and case-reason fields, now sanitised.

The third class concerned the confirmation flow. A weaker responder model could describe
a pending action as already completed; the evaluator, which would have detected this, was
being bypassed on confirmation turns. The confirmation prompt is now rendered
deterministically, the evaluator is no longer bypassed on mixed turns, natural
affirmatives such as 'yes, go ahead' now resolve a pending action correctly, and
duplicate escalations within a single turn are de-duplicated.

In each case the pattern is identical: a rule expressed only in the prompt proved
insufficient under pressure, and the correction was to enforce it in deterministic code,
accompanied by a regression test that reproduces the original failure."

==============================================================================
[16:30]  5. EVALUATION STRATEGY
==============================================================================
SHOW: the tests/ directory ; a pytest run ; graph.py::evaluator_node.

SAY:
"Evaluation operates at two levels. The offline level is the test suite, comprising
contract tests that pin the data schema and the policy figures, and a regression test
for every defect and guardrail described in this presentation. Every correction is
committed together with a test that reproduces the original failure, so that a resolved
defect cannot silently recur. The full suite presently passes in its entirety.

The online level is the in-flight evaluator, an LLM-as-judge that scores each reply
against the tool results and permits exactly one regeneration. The single-retry bound is
deliberate: a non-deterministic judge without a bound can oscillate indefinitely. The
bound is safe because the evaluator governs only response quality; the deterministic
guardrails constitute the safety floor, and remain effective even if the evaluator does
not."

[18:30]  7. KNOWN LIMITATIONS
==============================================================================
SAY:
"I will state the limitations plainly, as candour about the boundaries is consistent
with the operational posture I described at the outset.

First, and most importantly, the authentication and session model is a deliberate
simulation rather than a production mechanism. The session token is forgeable, the
password-reset code is static, registration does not verify ownership, and passwords use
an unsalted hash. The single highest-value production remediation is the replacement of
this scheme with opaque, server-issued session tokens behind an authenticating gateway.

Second, persistence uses JSON files with in-process locks; production would use a
relational database with transactions and idempotency keys, for which the repository
layer is already the clean seam.

Third, the post-execution guardrail runs after side effects have committed; production
would move the monetary checks fully pre-commit or adopt a compensating-transaction
pattern.

Fourth, language-model rate limits are an external constraint; a paid tier with
multi-provider failover and response caching is the remedy.

Fifth, the conversation state is in-memory and single-process; production would use a
persistent, shared store behind a load balancer."

SAY:
"To conclude, I return to the question with which I began: what happens when the AI is
wrong? The answer this system provides is: very little of consequence, because it was
never trusted to be correct about the matters that carry risk. The model proposes, and
deterministic code decides. You have seen configuration-driven policy, monetary safety in
three independent layers, an atomicity correction derived from anomalous production data,
deterministic guardrails for security and escalation, input and output sanitisation,
category-aware policy grounding, and a body of corrections produced by sustained
adversarial testing — each accompanied by a regression test. The limitations are real and
have been stated, each with a defined path to production. Thank you for your attention; I
would welcome any questions."
