1. OPENING & PROBLEM FRAMING
==============================================================================
SAY:
"Hello, and thank you for watching. This is my submission for AtlasCare, an
agentic AI customer-support system built for a retail company, Acme Retail Co.

Let me start with a question, because it's the question that shaped everything you're
about to see. A customer types one sentence — 'cancel my laptop and refund me forty
thousand rupees' — and an AI agent, on its own, decides whether to do it. Now: what
happens the moment that AI is wrong? Not if — when. When it misreads the order, when
it's been talked into something by a clever message, when the payment gateway times
out halfway through. Most chatbot demos never ask that question, because in a demo
the model is always right. I built this assuming it sometimes won't be.

So I didn't optimise for a demo that answers one question nicely. I optimised for
three in the morning, when something has gone wrong and someone is on call asking:
did we just refund money we shouldn't have, or quietly fail to refund money we
should have? Everything in this build is an answer to that one moment.

And the answer comes down to a single principle that you'll see again and again, so
watch for it — it's the spine of the whole project: the model gets to decide, but
code gets the final say. Behind every rule that touches money or changes an order
sits a deterministic guardrail, written in plain Python and fully tested, that does
not care whether the model obeyed its instructions. The prompt is the first line of
defence, and it's a good one — but it is never the last. Here's the part I find most
interesting, and I'll prove it to you as we go: nearly every one of those guardrails
exists because the prompt-only version proved insufficient under stress-testing. The
prototype was pushed deliberately, and it surfaced failure modes that weren't obvious
up front. This talk walks through those failure modes and the fix each one motivated."

2. ARCHITECTURE OVERVIEW   [scalable architecture]
==============================================================================
SHOW: a hand-drawn box diagram, OR the build_graph() function at the bottom of
      agent/graph.py (lines 2151-2178).

SAY:
"Let me start with the shape of the system, top to bottom.

The stack has three parts. There is a FastAPI application that exposes the HTTP
API — that's main.py. There is a Gradio web app that gives the customer a chat
window — that's gradio_app.py. And the core of the system, the actual agent, is a
LangGraph state machine that lives in agent/graph.py. The language model itself is
reached through an OpenAI-compatible client, and I want to flag that detail now
because it pays off later: because the client speaks the OpenAI protocol, I can
change the entire model provider with environment variables and no code change at
all. I'll demonstrate that.

The most important architectural decision is that a customer request does not run
through a free-wheeling loop where the model calls tools until it decides it's
done. It runs through an explicit, inspectable graph with named stages, and the
edges between those stages are themselves code I wrote. Here is the path a request
takes, and I'll name each node:

  input_redaction  →  confirmation_check  →  pre_guardrail  →  policy_grounding
                   →  tool_agent  →  tool_executor  →  post_guardrail
                   →  responder  →  evaluator  →  END

Let me walk each one, because each node exists for a reason.

The first node is input_redaction, and it is the true front door — it runs before
anything else on every turn. It deterministically masks sensitive data — payment
card numbers, CVVs, emails, and phone numbers — out of the customer's message
BEFORE that message enters the conversation history, reaches the language model, or
is written to any log. The reason it's first is compliance: this is a PCI and PII
hygiene control, and the system never needs a card number to service an order, so
the safest thing to do with one is to never let it travel any further than the
front door. I'll show the code for it in the security section.

The second node is confirmation_check. When a customer says 'yes' or 'go ahead',
that 'yes' only makes sense in the context of a question we asked on the previous
turn. This node runs first on every turn so that a pending 'are you sure?' from
last turn is resolved before anything else happens. It reads the stored pending
action and dispatches it if the user agreed.

The third node is pre_guardrail. This runs deterministic checks BEFORE the
language model ever sees the message: it catches safety and fraud reports that
must be escalated, refund amounts over our limit, malformed order IDs, empty
messages, and oversized messages. The key word is 'before' — these decisions are
made without trusting the model at all.

The fourth node is policy_grounding, and this is where the architecture makes an
explicit split that I want to flag up front, because it's a deliberate enterprise
pattern: question-answering and action-taking go down two SEPARATE branches. If the
customer is asking a GENERAL policy question — 'what's your refund policy', 'how long
is the return window' — this node does document retrieval against the knowledge base
and routes the request straight to the responder with those articles as grounding,
SKIPPING the planner and the tools entirely. That's the Q&A-via-RAG branch. Anything
that names an order or asks for an action falls through to tool_agent — the
Action-via-Tools branch. The reason for the split is two-fold: trust, because a policy
answer is grounded in the articles we actually published rather than improvised from
the model's priors; and cost, because a general question is answered in one grounded
call and never pays for the planner or a tool loop. I'll show this in its own section.

The fifth node is tool_agent. This is the planner: the language model looks at the
conversation and chooses which tools to call. Here I run a two-tier model strategy
that is a direct cost-and-latency lever. A strong, large model handles complex,
mutating, or filtering queries where reasoning matters; a small, fast model
handles simple lookups. The effect is that 'where is my order' comes back in under
about three seconds on the cheap model, while 'cancel the laptop and refund me'
gets the powerful model. I'll explain how the routing decides which is which.

The sixth node is tool_executor. This actually runs the tools the planner chose,
and this is where several of my code-enforced policies live — it is the chokepoint
through which every action must pass, so it's the right place to enforce things.

The seventh node is post_guardrail. This is a final money-safety net that runs AFTER
the tools have executed, to catch the catastrophic case where money was disbursed
that should not have been.

The eighth node is responder. This writes the actual customer-facing reply, and
critically it writes that reply only from the verified tool results, never from
the customer's unverified claims.

The ninth node is evaluator. This is an LLM-as-judge pass that scores the reply.
If the reply is weak, it sends it back for exactly one regeneration, and then it
ships whatever the second attempt produced. That single-retry cap is a deliberate
bound — I'll explain why one and only one.

Underneath the graph there is a clean layering: tools call services, services call
repositories, and repositories read and write the data files. The four canonical
data files — orders, CRM cases, knowledge-base articles, and payment config — are
the source of truth. Everything else, the user accounts, the session mappings, the
refund ledger, and the audit log, is DERIVED from those four files at startup, so
that a fresh data folder boots cleanly without any manual setup. And one detail
that matters for correctness: money is handled with Python's Decimal type, never
floating point, so we never get a rounding error on a rupee amount."

SHOW: observability/tracer.py and observability/trace_store.py, then admin_dashboard.py.

SAY:
"Observability is first-class, not an afterthought, because you cannot operate
what you cannot see. Every single request is assigned a trace ID the moment it
arrives — you can see that in tracer.py, the format is 'trc-' followed by twelve
hex characters, short enough to fit in a log line and unique enough to correlate
in production. As the request flows through the graph, every tool call is recorded
into that tracer with its status and its latency, and every guardrail trigger is
recorded too. At the end, the whole trace is written into a thread-safe in-memory
ring buffer — trace_store.py — that holds the last five hundred requests.

That store is what powers the admin dashboard, and it also computes KPIs directly:
total requests, escalation rate, guardrail hits, ownership-denial count, error
count, and latency percentiles including a count of SLA breaches over three
seconds. So at any moment an operator can answer 'how is the system behaving, how
often is it escalating, and is it meeting its latency target' — which is exactly
what you need to be on call for this."

3.2  MONEY SAFETY — THE REFUND THRESHOLD, IN THREE LAYERS   [business KPI · compliance]

"The single worst thing this system can do is autonomously pay out a refund that

depth: a single bug, or a single clever prompt, should never be enough to open the
money tap. Let me walk each layer's code.

3.3  CANCEL / REFUND ATOMICITY — NEVER CANCEL WITHOUT A REFUND TRACE   [business KPI]
------------------------------------------------------------------------------
SHOW: agent/graph.py::_handle_cancel_item, the ATOMICITY GUARANTEE block (lines 1088-1190).

SAY:
"This one is worth dwelling on, because it surfaced not from reading code but from
inspecting the live data the agent had produced.

One order in the data had three cancelled items but only two refund records. One item
had been cancelled with no refund on record anywhere. For a system that handles money,
that is the unrecoverable failure: the customer lost an item and got nothing back, and
there's no trace to even find it.

The root cause is an ordering one. The handler was doing the obvious thing: it
cancelled the item first, and then it processed the refund. But if the gateway call was
declined, or the session dropped in between those two steps, the cancellation was
already committed and the refund was simply lost.

The fix is a transaction-ordering fix, and you can see it in _handle_cancel_item. The
order of operations is reversed: secure the refund TRACE first, commit the
cancellation second. Let me walk the block.

WALK THE CODE — graph.py::_handle_cancel_item, the ATOMICITY GUARANTEE block, lines 1088-1190:
"Read the comment on lines 1088-1094 first — it states the invariant: a cancellation
must never be committed without a durable refund trace.
  - Lines 1095-1098: three variables are set up — escalated, refund, refund_case_id —
    and the method is normalised. Nothing has been committed yet.
  - Lines 1100-1108: the COD branch. Cash-on-delivery has no electronic source to
    refund to, so we never auto-disburse — we create a specialist case, which IS the
    durable trace, and mark escalated.
  - Lines 1109-1111: the normal branch — it calls _payment.process_refund, which
    returns an actual disbursement record. That record is the trace.
  - Lines 1112-1120: if process_refund raises RefundThresholdError — the layer-two
    guard from 3.2 firing on a high-value amount — we can't auto-pay, so we file a
    specialist case as the trace and mark escalated. Still a trace.
  - Lines 1121-1136: this is the critical branch. If the refund fails for ANY other
    reason — gateway down, session dropped — we log it and RETURN immediately with an
    error, having touched nothing. The order is left completely intact. No trace, no
    cancel. That is the safe failure.
  - Lines 1138-1140: only now, with a trace guaranteed to exist, do we call
    _oms.cancel_item to actually commit the cancellation.
  - Lines 1141-1162: and even here there's a belt-and-braces case — if the cancel
    itself somehow fails after the refund succeeded, we log CANCEL_AFTER_REFUND_FAILED
    and write a 'refund_without_cancel' audit record. That's NOT a money gap — the
    money is safely recorded and the item is simply still active, which a human can
    reconcile.
  - Lines 1164-1190: the success path attaches the refund or case to the result and
    writes the audit entry.
The whole point is the ordering: trace on lines 1100-1137, cancellation on 1138-1140,
never the reverse.

SAY:
"So think about the worst case now. It has degraded from 'item cancelled, money gone,
no record' — which is unrecoverable — to 'refund recorded but the item is still
active' — which is fully recoverable, because the record exists and a human can
finish the job. That is transaction-ordering thinking applied to an agent, and it
came directly from staring at anomalous data."

SHOW: graph.py::_handle_cancel_item, the order-status guard (around line 1112), and
      _CANCELLABLE_STATUSES (around line 820).

SAY:
"And here is the follow-up bug, worth showing precisely because it illustrates how one
fix can open another hole — which is the discipline the whole talk is about. Re-running
the battery after the atomicity reorder surfaced it: cancelling an item on an order that
had already SHIPPED. A shipped order can't be cancelled — that rule lives inside the OMS
tool's cancel_item. But remember what the atomicity fix did: it now secures the refund
FIRST and commits the cancel SECOND. So the order of events became: disburse the refund,
and only THEN call cancel_item, which correctly refused — leaving a refund paid out for
an item that was never cancellable. The reorder that fixed one money gap had quietly
created another.

The fix is a deterministic pre-check. Read _handle_cancel_item: before it secures any
refund trace, it now checks 'order status in _CANCELLABLE_STATUSES' — the frozenset
{placed, processing}, defined once near the top of the module so the handler and the
confirmation classifier can't drift. If the order isn't cancellable, it returns a
clean error immediately and touches nothing — no refund, no mutation. The status
rule used to live ONLY inside the tool that runs last; now it's enforced up front,
before any money can move. Exemplar query to show this live — ask to cancel the Nike
sneakers on ORD-10003, which has already shipped: the bot refuses cleanly, and the
trace shows no refund was disbursed. There's a regression test sitting behind it,
test_cancel_non_cancellable_order_does_not_refund, asserting exactly that: no refund
record for a shipped order.

There's a second, smaller bug from that same re-run worth one sentence, because it's
about honesty in the UI. On a request that staged a confirmation alongside a
completed action, the system flagged the whole turn as 'resolved' while a confirmation
was still pending. The tool_executor now forces task_complete to False whenever it
stages a confirmation — a pending question is by definition an unresolved turn — so
the interface never tells the customer 'done' while it's still waiting on their 'yes'."

3.8  THE RAG BRANCH — RETRIEVAL FOR Q&A, TOOLS FOR ACTIONS   [scalable architecture · compliance · KPIs]
------------------------------------------------------------------------------
SHOW: graph.py::policy_grounding_node (around line 1738), _detect_policy_query
      (around line 754), and _route_policy_grounding (around line 1771); then the two
      conditional edges in build_graph that form the fork.

SAY:
"The last piece in the request path is the retrieval branch, and it's an explicit
architectural split I want to be clear about, because it's a standard enterprise
pattern: question-answering and action-taking are routed down two SEPARATE branches.
A document-retrieval — RAG — branch handles 'what is your refund policy'-style
questions; the tools branch handles anything that changes an order. The risk it
addresses is specific: a language model asked a policy question will happily produce a
confident, fluent, and possibly WRONG answer from its own training priors, and for a
support bot inventing policy is its own kind of harm — a customer acts on it. So
general policy questions don't go to the model's imagination; they go to the knowledge
base, and they never touch the planner or the tools at all.

WALK THE CODE — graph.py::_detect_policy_query, around line 754:
"This deciding function is deliberately conservative. It returns knowledge-base tags
ONLY when the message is a GENERAL policy question — there's a policy cue like
'policy', 'window', 'how long', and a topic like refund, return, warranty, or
cancellation. And it returns nothing — hands back to the normal planner — the moment
the message names a specific order ID or asks for a transactional action. So 'what's
the return window' is grounded; 'cancel item 1 on ORD-10002' is not. It never hijacks
a real action.

WALK THE CODE — policy_grounding_node and _route_policy_grounding, around lines 1738 and 1771:
"When tags come back, the node retrieves the matching articles from the KB and builds
a compact grounding block — the official titles and text — and stamps a 'kb_grounding'
event into the tracer. Then the router does something I think is the elegant part: for
a pure policy question it routes STRAIGHT to the responder, skipping the planner model
entirely. The responder then writes the answer using ONLY those retrieved articles,
quoting the real figures — the twenty-five-thousand limit, the thirty-day window, the
five-business-day SLA — straight from the files we actually ship. For a mixed turn —
policy plus an action — the grounding is prepended to the responder's context instead,
so the action still runs but the policy part stays anchored. And the grounding field
is reset every turn, so a stale answer can never leak into a later, unrelated request.

SAY:
"Three things to call out. First, the split is at the ROUTING layer, not buried inside
the executor — the fork is the conditional edge out of policy_grounding: a pure policy
question goes to the responder, everything else goes to tool_agent. You can see that
branch in the graph diagram on the admin dashboard.

Second, it's a deliberately HYBRID design, not a hard either-or. The explicit RAG
branch handles the clean, high-volume case — a general policy question — for free. But
search_kb is ALSO still available as a tool inside the executor, so an order-SPECIFIC
or mixed question — 'what's the return window on ORD-10005, and cancel item two' — can
still pull the knowledge base on the action path, and the same grounding is prepended
to the responder's context there. The conservative classifier only takes the RAG
branch when there's no order ID and no action verb, precisely so a Q&A path can never
hijack something that moves money.

Third, this is retrieval-augmented generation in its honest form — the model phrases
the answer, but the FACTS come from our data, not its memory, which is the
config-as-source-of-truth principle from section 3.1 applied to the customer-facing
words — and it's a KPI win: a general policy question is one grounded call with no
planner round-trip, so it's faster and cheaper.

To show it live, two exemplar queries side by side. First, 'What is your refund
policy?' — the trace shows a 'kb_grounding' step and NO planner call, and the answer

SAY:
"First, and most importantly, the authentication and session model is a deliberate
simulation, not a production auth system, and I want to be very explicit about its
weaknesses because they are real.

  - The session token is forgeable. The session store will resolve any string that
    embeds a customer ID pattern — for example 'sess-CUST001-anything' — straight to
    that customer. The random part of a real token is never actually verified, so
    the token carries no secret. In effect, anyone who knows a customer-ID format
    can impersonate a customer by hitting the query endpoint directly. The reason
    this exists is that the assignment explicitly called for a pragmatic
    session-to-customer simulation rather than a real login system, and the embedded
    pattern lets every customer be exercised in testing without pre-provisioning
    each session. The production fix is to make sessions opaque, random,
    server-issued tokens stored server-side, and to delete the embedded-pattern
    fallback entirely so that only an explicitly issued session ever resolves.

  - Password reset uses a single static OTP of 9999, and that OTP is even returned
    in the API response. Anyone could reset anyone's password. The reason is, again,
    demo scope — there is no email or SMS provider wired up. The production fix is a
    real one-time-passcode generated server-side, delivered out of band, time-boxed,
    single-use, and of course never returned in the response body.

  - Registration lets a caller bind their new account to any customer ID they like,
    with no proof of ownership, so a user could claim another customer's identity.
    The production fix is to verify ownership of the customer record at registration
    — for example via a verified email match — and to reject a customer ID that's
    already bound.

  - Passwords are hashed with unsalted SHA-256, and the derived demo accounts all
    share the password 'password'. The production fix is a slow, salted password
    hash — bcrypt, scrypt, or Argon2 — and forced credential rotation on first
    login.

  - There is no rate limiting on the login, OTP, or reset endpoints, and the admin
    and internal data endpoints have no authentication on them at all. The
    production fix is to put this behind an API gateway that enforces authentication
    and per-tenant rate limiting, and to require constant-time secret comparison
    everywhere.

  The honest summary is that the root cause of most of these is one design choice —
  that the session ID encodes the customer identity in plaintext and is trusted as
  proof of it — and the single highest-value production fix is to replace that with
  opaque server-issued sessions. I flag all of this clearly because an agent that can
  refund money to the wrong person must never ship with this auth model outside of a
  sandbox.

Second, persistence is JSON files with in-process write locks. The reason is
assignment scope and fast iteration. The production fix is PostgreSQL with real
transactions, row-level locks, and idempotency keys — and the repository layer is
already the clean seam to swap a database in behind, with no change to the business
logic above it.

Third, the post-execution guardrail, GR-004, runs after side effects have already
committed. The reason is that some checks genuinely need the actual result to
inspect. The production fix is to move the money checks fully pre-commit, or to
adopt a saga or compensating-transaction pattern so that a blocked reply
automatically triggers a reversal. Today, at minimum, we log every committed action
at critical severity for reconciliation, and in practice GR-004 is nearly
unreachable because the payment tool blocks an over-limit refund at the source.

Fourth, language-model rate limits are a hard external ceiling. The free tier was
capped — Gemini's free tier is about twenty requests a day, which interrupted an
adversarial test battery mid-run. The reason is simply free-tier provider quotas. The production fix is a paid tier plus the multi-provider
failover the OpenAI-compatible client already makes trivial, plus a request queue
and response caching for repeated reads.

Fifth, some quality still depends on the model — narration completeness, slot
accuracy — even though every critical decision is code-enforced. The reason is that
it is, at the end of the day, a language model. The production fix is tighter
structured extraction for the high-stakes intents, plus continuous online evaluation
metrics with alerting and A/B testing on any prompt or model change.

Sixth, the state — the checkpointer and the trace store — is in-memory and
single-process. The reason is simplicity. The production fix is a Redis-backed or
otherwise persistent checkpointer, horizontal scaling behind a load balancer, and
the API gateway I mentioned for authentication, authorization, and rate limiting."

SAY:
"So let me bring it back to the question I opened with — what happens when the AI is
wrong? The answer this project gives is: not much, because it was never trusted to
be right about the things that matter. The model proposes, and code gets the final
say. You saw config-driven policy so an operator can change a rule without a deploy;
refund safety in three independent layers; a cancel-and-refund ordering fix that came
straight out of an anomalous row in the live data; deterministic guardrails for
security, escalation, and confirmation, added precisely because the prompt-only
versions proved insufficient under testing; retry and backoff so a hiccup never reaches
the customer; full tracing and an audit trail so nothing the agent does is invisible;
and a regression test sitting behind every single fix, so none of those bugs can ever
come back. The limitations are real and I've stated them plainly, each with a concrete
path to production. Thank you very much for watching — I'd genuinely enjoy talking
through any part of it."
